# /study-note

대상 파일: `$ARGUMENTS` (비어 있으면 현재 IDE에서 열려 있는 `docs/` 하위의 `.md` 파일을 대상으로 한다)

내용을 이 md 파일에 막 붙여넣은 상태라고 가정하고, 아래 4단계를 순서대로 수행한다.
각 단계가 끝나면 다음 단계로 넘어가기 전에 무엇을 했는지 한두 문장으로 보고한다.

---

## 1단계 — 내용 검수 (전체적으로 맞는 내용인지 다듬기)

- 기술적으로 틀리거나 애매한 문장을 찾아 수정한다 (예: 시간/공간 복잡도 오기, 순서 오류, 근거 없는 단정).
- 같은 개념을 설명하는 문단들끼리 모순되지 않는지 확인한다.
- **문장을 다듬되 저자의 어투와 문단 길이 스타일은 유지한다.** 전체를 새로 쓰지 않는다.
- 이 저장소의 노트는 다음과 같은 항목형 구조를 공통으로 쓴다 (참고용 — 파일마다 없는 항목이 있을 수 있고, 강제로 끼워 맞추지 않는다):
  `핵심 요약 → 등장 배경 → 핵심 개념 → 구조와 동작 원리 → 코드/사용 예시 → 성능 특성 → 장점과 단점 → 사용 기준 → 비슷한 개념 비교 → 실무 적용 → 자주 하는 오해 → 면접 답변 → 예상 면접 질문 → 추가 학습 방향 → 최종 체크리스트 → 한 줄 결론`
- 코드 블록의 언어 태그(````java`, ````sql` 등)와 예시 코드가 실제로 컴파일/실행 가능한 수준인지 확인한다.

## 2단계 — 도식 자료 만들기

- 본문에서 그림이 있으면 이해가 확 쉬워질 지점을 1~3곳 고른다. 흔한 후보:
  - 여러 대상의 증가/감소 추세 비교 (예: 복잡도 곡선)
  - 단계가 있는 과정 (예: 이진 탐색처럼 범위가 줄어드는 과정)
  - 구조 다이어그램 (트리, 연결 리스트, 요청 흐름 등)
- 반드시 **SVG를 Python 스크립트로 생성**한다 (그림판/외부 이미지 생성 금지 — 재현 가능하고 텍스트로 diff 가능해야 함). `docs/.../<노트폴더>/gen_*.py`가 아니라 임시 스크립트는 스크래치패드에 만들고, 결과 `.svg`만 노트와 같은 폴더에 저장한다.
- SVG 작성 규칙 (기존 `bigo-growth.svg`, `binary-search.svg` 참고):
  - 배경은 밝은 카드(`#fcfcfb` 계열) + 옅은 테두리 — 다크/라이트 어느 Obsidian·브라우저 테마에서도 안정적으로 보인다.
  - 폰트는 `system-ui, -apple-system, "Segoe UI", "Malgun Gothic", sans-serif` (Google 폰트 등 외부 폰트 금지 — 오프라인·GitHub Pages 양쪽에서 깨지지 않게).
  - 계열별 색은 고정 순서로 배정하고, 색만으로 구분하지 말고 직접 라벨을 붙인다 (색약 대응).
  - 만든 뒤 **반드시 `scripts/preview_svg.py`로 PNG 변환 후 Read 도구로 직접 눈으로 확인**한다. 라벨 겹침, 잘림, 어색한 배치가 있으면 좌표를 고쳐 다시 렌더링한다.
- 노트 본문에는 표준 마크다운 임베드로 넣는다: `![설명](파일명.svg)` — Obsidian과 GitHub Pages(mkdocs) 양쪽에서 그대로 렌더링된다. 이미지 바로 아래에 한 줄짜리 캡션(이탤릭)을 붙인다.
- 기존 문단 구조(형식)는 건드리지 않는다 — 이미지+캡션만 해당 위치에 삽입한다.

## 3단계 — 형식·배치 유지

- 노트 파일은 `docs/<NN-주제>/<소주제>/<파일명>.md` 경로에 있어야 한다 (예: `docs/02-정렬/퀵정렬/퀵정렬.md`). 파일명은 폴더명과 동일하게 한다 (Obsidian 빠른 전환 시 구분을 위해 — `Main.md` 금지).
- 이미지(svg)는 노트와 **같은 폴더**에 둔다.
- `mkdocs.yml`의 `nav:`에 새 노트를 추가한다 (없으면 새 항목 만들기, 있으면 그대로 둠). 기존 항목의 순서/이름은 임의로 바꾸지 않는다.
- `docs/index.md`의 목차 표/목록에도 새 노트 링크를 한 줄 추가한다.
- 새 최상위 주제(`NN-주제`)를 새로 만드는 경우, 번호는 기존 폴더들과 겹치지 않게 다음 순번을 쓴다.

## 4단계 — GitHub Pages 배포 준비

1. `mkdocs build --strict` 실행 — 깨진 링크·이미지·nav 참조 오류를 잡는다. 실패하면 원인을 고치고 다시 빌드한다.
2. (도식을 추가했다면) `mkdocs serve`로 로컬 서버를 띄우고, `scripts/preview_svg.py`가 아니라 headless 브라우저로 실제 페이지를 스크린샷해서 레이아웃이 깨지지 않았는지 확인한다. 확인 후 서버는 반드시 종료한다.
3. `git status`로 변경 파일을 확인하고, `git diff`로 의도치 않은 변경(예: `.obsidian/workspace.json`, `site/` 폴더)이 섞이지 않았는지 점검한다.
4. 변경 요약(새/수정된 파일 목록, 추가한 도식 설명)을 사용자에게 보여주고 **커밋/푸시 여부를 확인받는다.** 확인 전에는 push하지 않는다. (승인 시: `git add -A && git commit -m "..." && git push`)

### 참고: SVG 미리보기 렌더링

```
python scripts/preview_svg.py "docs/경로/파일.svg"
```
같은 폴더에 `파일.png`가 생성된다. Read 도구로 열어 확인한 뒤, 확인용 PNG는 커밋하지 않는다 (`.gitignore`에 `*.preview.png` 등으로 제외하거나 스크래치패드에 생성).
